# Introducing Tanstack Query

- Our app currently uses the `useState` and `useEffect` hooks to fetch and display reviews. Although this approach works, it has several limitations.

### Problems with `useState` + `useEffect`

#### 1. No Automatic Retries

- If the API request fails, the error is shown to the user immediately.
- There are no automatic retry attempts.
- Automatic retry logic can be implemented manually, but it requires additional code.

#### 2. No Built-in Caching

- If the user navigates away from the page and later returns, the reviews are fetched again.
- Previously fetched data is not cached.
- Caching can also be implemented manually, but it requires extra work.

## TanStack Query

TanStack Query solves these problems and provides several powerful features.

### Benefits

- Automatic retries for failed requests.
- Built-in caching.
- Cleaner and more concise code.

## Installing TanStack Query

Open a terminal in the client directory and install the package:

```bash
bun add @tanstack/react-query
```

## Setting up the Query Client

### Step 1: Import Required Objects

In `main.tsx`, import:

```tsx
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
```

### Step 2: Create a Query Client

```tsx
const queryClient = new QueryClient();
```

### Step 3: Wrap the Application

Wrap the entire application inside `QueryClientProvider`.

```tsx
<QueryClientProvider client={queryClient}>
    <App />
</QueryClientProvider>
```

The `client` prop receives the `queryClient` object created earlier.

## Refactoring the Review List Component

### Remove State Variables

Previously, separate state variables were declared for:

- Review data
- Loading state
- Error state

These are no longer necessary because TanStack Query manages them.

### Using the `useQuery` Hook

```tsx
const {
    data: reviewData,
    isLoading,
    error,
} = useQuery<GetReviewsResponse>({
    queryKey: ["reviews", productId],
    queryFn: () => fetchReviews(),
});
```

- `useQuery()` is a generic function, so we have to specify the generic type argument.
- The generic type `GetReviewsResponse` specifies the type of data returned from the backend.

### Understanding `queryKey`

```tsx
queryKey: ["reviews", productId];
```

- `queryKey` is used for caching.
- This key allows TanStack Query to locate cached data quickly.

#### Using Dynamic Values

- In a real application, every product has different reviews.
- If only `"reviews"` is used, then every product would share the same cache entry.
- Adding `productId`, creates a unique cache entry for each product.

### Understanding `queryFn`

- It is an arrow function responsible for fetching data.

```tsx
queryFn: () => fetchReviews(),
```

### Values Returned by `useQuery`

- The hook returns an object containing several useful properties.

Important ones include:

```tsx
const {
    data: reviewData,
    isLoading,
    error,
} = useQuery(...)
```

Here:

- `data` is renamed to `reviewData`.
- Existing code continues to work without modification.

### Simplifying `fetchReviews()`

With TanStack Query:

- Loading is handled automatically.
- Errors are handled automatically.

Therefore, remove:

- `try...catch`
- `finally`
- Calls to update loading state

```tsx
const fetchReviews = async () => {
    const { data } = await axios.get<GetReviewsResponse>(
        `/api/products/${productId}/reviews`,
    );
    return data;
};
```

The function only:

1. Calls the backend.
2. Returns the response data.

### Removing `useEffect`

Previously:

```tsx
useEffect(() => {
    fetchReviews();
}, []);
```

- This is no longer needed because `useQuery` automatically performs the fetch.
- The entire `useEffect` can be deleted.

### Render Error

```tsx
if (error) return <p className="text-red-500">{error.message}</p>;
```

OR

```tsx
if (error)
    return (
        <p className="text-red-500">Could not fetch the reviews. Try again!</p>
    );
```

## Result

1. Loading skeletons appear.
2. Error is **not** shown immediately.
3. TanStack Query automatically retries the request.

By default:

- It retries **3 times**.

Only after all retries fail does the error appear.
