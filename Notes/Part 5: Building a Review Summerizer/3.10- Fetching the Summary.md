# Fetching the Summary

## Goal

Enhance the **Get Reviews** endpoint so that it returns both:

- The list of reviews
- The review summary

This allows the frontend to fetch everything it needs with a **single API call**, making the client-side code:

- Cleaner
- More efficient
- Simpler to maintain

## Removing an Unnecessary Service Layer

- The `getReviews()` method inside the review service is only forwarding the request to the repository.
- There is **no business logic**.

```ts
async getReviews(productId: number): Promise<Review[]> {
    return reviewRepository.getReviews(productId);
}
```

### Why remove it?

- A service layer should contain business logic.
- If it only forwards calls without adding any value, it becomes unnecessary.

### Refactoring

Delete:

- `getReviews()` from `reviewService`
- Unused `Review` import/type

Then make the controller call the repository directly.

Instead of:

```ts
const reviews = await reviewService.getReviews(productId);
```

Use:

```ts
const reviews = await reviewRepository.getReviews(productId);
```

## Fetching Both Reviews and Summary

Inside the controller:

```ts
const reviews = await reviewRepository.getReviews(productId);
const summary = await reviewRepository.getReviewSummary(productId);
```

Return:

```ts
res.json({ summary, reviews });
```

Now the frontend receives both pieces of information together.

## Initial Attempt to Handle Expired Summaries

The controller initially checked whether the summary had expired.

Example logic:

```ts
if (summary && summary.expiresAt > new Date()) {
    return summary.content;
}

return null;
```

### Problem with This Approach

- The expiration logic now exists inside the controller.
- However, the **same logic already existed elsewhere**.

Specifically inside:

```
reviewService.summarizeReviews()
```

It also checked:

- Does a summary exist?
- Has it expired?

This means the same business rule is duplicated.

### Correct Responsibility

- Expiration checking belongs in the **repository** (data access layer), not in controllers or services.

The repository should only return:

- A valid summary
- Or `null`

The caller should not need to know how expiration works.

## Moving Expiration Logic into the Repository

Update repository:

```ts
async getReviewSummary(productId: number): Promise<string | null> {
    const summary = await prisma.summary.findFirst({
        where: {
            AND: [{ productId }, { expiresAt: { gt: new Date() } }],
        },
    });
    return summary ? summary.content : null;
}
```

Meaning:

Return a summary where:

- Product ID matches
- Expiration time is greater than the current time

### Prisma Comparison Operators

Prisma provides comparison operators instead of direct values.

Example:

```ts
expiresAt: {
    gt: new Date();
}
```

Prisma translates this into SQL similar to:

```sql
WHERE expiresAt > CURRENT_TIMESTAMP
```

instead of

```sql
WHERE expiresAt = CURRENT_TIMESTAMP
```

which would be incorrect.

### Why `findUnique()` Cannot Be Used

- After adding multiple filters with `AND`, the query is no longer based solely on a unique constraint.
- So, replace `findUnique()` with `findFirst()`.

### Avoid Exposing Internal Database Details

The `summary` model contains:

- id
- productId
- content
- generatedAt
- expiresAt

Only one of these is actually useful outside the repository:

```
content
```

The others are implementation details.

Clients should not know:

- database IDs
- expiration timestamps
- generation timestamps

This keeps the repository abstraction clean.

## Handling Invalid Product IDs

Example:

```
GET /products/10/reviews
```

Returning:

```json
{
    "summary": null,
    "reviews": []
}
```

is misleading because the product itself is missing.

A more RESTful response is:

```
404 Not Found
```

### Validating Product Existence

Before fetching reviews:

```ts
const product = await productRepository.getProduct(productId);

if (!product) {
    res.status(404).json({
        error: "Product does not exist.",
    });
    return;
}
```

Only if the product exists should the controller continue fetching:

- reviews
- summary
