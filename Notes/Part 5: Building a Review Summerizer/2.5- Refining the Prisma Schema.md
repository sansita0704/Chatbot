# Refining the Prisma Schema

## 1. Table Names

- By default, Prisma uses the model name as the table name.
- Since model names usually follow **PascalCase** (e.g., `Product`), the generated table name will also be `Product`.
- However, database tables are commonly named using **lowercase plural names**, such as `products`.
- We can override the default table name using the `@@map` attribute.

```prisma
@@map("products")
```

The model will still be called `Product` in our code, but the actual table in MySQL will be named `products`.

## 2. Database Data Types

- Prisma automatically maps `String` data type to `varchar(191)`.
- If we want more control over the database column type, we can use the `@db` attribute.

Examples:

```prisma
@db.VarChar(255)
@db.Text
@db.TinyInt
```

### `@db.Text`

- Creates a `TEXT` column.
- It provides significantly larger storage than `VARCHAR`, making it suitable for storing long review content.

### `@db.TinyInt`

- A normal integer can store values up to billions, whereas ratings typically range from **1–5**.
- So, for fields like ratings, we don't need such a large range.
- `TinyInt` can store numbers up to 255.

## Creating a New Migration

After making changes to the Prisma schema, we need to create a new migration so that the database reflects those changes.

Run:

```bash
bunx prisma migrate dev
```

- Prisma will ask for a migration name.
- Give it a descriptive name based on the changes made.
