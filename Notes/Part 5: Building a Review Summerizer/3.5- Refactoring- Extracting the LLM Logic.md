# Extracting the LLM Logic

## Problem with the Current Implementation

- The application currently has **multiple direct references to Groq** across different services (e.g., `reviewService` and `chatService`).

Example issues:

- Groq client is instantiated in multiple places.
- Groq-specific APIs (`client.responses.create`, `input`, `max_output_tokens`, etc.) are used directly inside services.

### Why is this a problem?

If the project switches from **Groq** to another provider like **Gemini** or **Claude**, several changes would be required:

- Replace every Groq client instance.
- Rewrite provider-specific API calls.
- Update arguments and response handling throughout the codebase.

This violates the principle of keeping provider-specific implementation isolated.

## Solution: Create an LLM Module

Create a dedicated directory:

```
server/
└── llm/
    └── client.ts
```

- This module will become the **single place responsible for communicating with any LLM provider.**
- All provider-specific logic will remain inside this module.
- The rest of the application will interact only with a clean, provider-agnostic interface.

### Move Groq Client into the Module

Move the Groq client creation code from the services into `llm/client.ts`.

```ts
const client = new Groq({
    apiKey: process.env.GROQ_API_KEY,
});
```

Now every service uses this single client indirectly.

### Export an LLM Client Object

Follow the same pattern used for controllers, services, and repositories.

```ts
export const llmClient = {
    generateText() {},
};
```

This object exposes only one public method `generateText()`.

### Create a GenerateTextOptions Type

Instead of exposing Groq-specific arguments, define a custom **application-specific type**.

Example:

```ts
type GenerateTextOptions = {
    model?: string;
    prompt: string;
    temperature?: number;
    maxTokens?: number;
};
```

### Create GenerateTextResult Type

Define another application-specific type:

```ts
type GenerateTextResult = {
    id: string;
    text: string;
};
```

This type is independent of any LLM provider.

### Assign Default Values

- Inside `generateText()`, destructure the options object.

Assign default values:

```js
async generateText({
    model = 'llama-3.1-8b-instant',
    messages,
    temperature = 0.2,
    maxTokens = 300,
}: GenerateTextOptions): Promise<GenerateTextResult>
```

### Groq API Call

Inside `generateText()`:

```ts
const response = await client.chat.completions.create({
    model,
    messages,
    temperature,
    max_completion_tokens: maxTokens,
});
```

### Return

```ts
return {
    id: response.id,
    text: response.choices[0]?.message.content || "",
};
```

All Groq response processing remains hidden inside the LLM module.

### Benefits of This Design

Services depend only on:

```ts
llmClient.generateText(...)
```

They do **not** know:

- Groq exists
- Gemini exists
- Claude exists
- How requests are sent
- How responses are formatted

If the provider changes in the future, only **`llm/client.ts`** needs modification.
The services remain unchanged.

## Refactor Review Service

```ts
const response = await llmClient.generateText({
    messages: [
        {
            role: "user",
            content: prompt,
        },
    ],
    maxTokens: 500,
});

return response.text;
```

## Refactor Chat Service

Perform the same cleanup.

Remove:

- Groq client
- Groq imports
- Groq-related comments

Replace direct Groq calls with:

```ts
const response = await llmClient.generateText({
    messages: [systemMessage, ...chatHistory],
});

chatHistory.push({
    role: "assistant",
    content: response.text,
});

return { id: response.id, message: response.text || "" };
```
