# Extracting Routes

- So far, all of our route definitions are inside `index.ts`.
- This is fine for small applications.
- However, as the application grows, keeping every route inside a single file quickly becomes difficult to maintain.

A better approach is to move route definitions into a dedicated module.

## Create `routes.ts`

Create a new file:

```text
server/routes.ts
```

```ts
import express from "express";
import type { Request, Response } from "express";
import { chatController } from "./controllers/chat.controller";

const router = express.Router();

// Define a route
router.get("/", (req: Request, res: Response) => {
    res.send("Hello World!");
});

router.get("/api/hello", (req: Request, res: Response) => {
    res.json({
        message: "Hello World!",
    });
});

router.post("/api/chat", chatController.sendMessage);
```

### Why Use a Router?

Previously, we registered routes on the Express application:

```ts
app.get(...)
```

```ts
app.post(...)
```

- Inside `routes.ts`, we no longer work with the `app` object.
- Instead, we create a **Router**.

```ts
const router = express.Router();
```

- On this router, we register our endpoints.

> A router is simply a mini Express application that is responsible only for defining routes.

Instead of:

```ts
app.get(...)
```

we now write:

```ts
router.get(...)
```

### Route Handlers

Currently we have:

```ts
router.get("/api/hello", (req: Request, res: Response) => {
    res.json({
        message: "Hello World!",
    });
});
```

- In a real application, this is generally **not recommended**.
- Instead, routes should simply reference controller methods.

Example:

```ts
router.get("/api/hello", helloController.getMessage);
```

However, in this course the above route is only a simple one-line demo endpoint, so there is no need to refactor it.

### Export the Router

At the end of `routes.ts`:

```ts
export default router;
```

- We use a **default export** because this module exposes only one thing: the router

### Update `index.ts`

- Previously, `index.ts` contained all route definitions.
- Now it simply creates the application and registers the router.

```ts
const app = express();
app.use(express.json());
app.use(router);
```

1. Create the Express application.
2. Register the JSON parsing middleware.
3. Register the router.

## Final `index.ts`

```ts
import express from "express";
import dotenv from "dotenv";
import router from "./routes";

dotenv.config();

// Create the app
const app = express();

// Register middleware
app.use(express.json());

// Register routes
app.use(router);

// Initialize the port
const port = process.env.PORT || 3000;

// Start the server
app.listen(port, () => {
    console.log(`Server is running on http://localhost:${port}`);
});
```

### Project Structure

- `routes.ts`: contains all endpoints.

As the project grows, we will typically split routes by feature.

Example:

```text
routes/
│
├── chat.routes.ts
├── product.routes.ts
├── order.routes.ts
├── admin.routes.ts
└── auth.routes.ts
```
