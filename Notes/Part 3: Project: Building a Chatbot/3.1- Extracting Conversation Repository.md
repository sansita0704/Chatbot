# Conversation Repository

One important rule of the layer architecture is that **dependencies always flow from top to bottom**.

## Dependency Direction

```text
Controllers
      ↓
Services
      ↓
Repositories
```

This means:

- Controllers can communicate with Services.
- Services can communicate with Repositories.
- Repositories should **not** communicate with Services.
- Services should **not** communicate with Controllers.

Because repositories are at the bottom, they are the **most fundamental layer** of our application.

## Repositories

- A repository is responsible for **data access**.

### Data Access in Our Application

Currently, we have:

```ts
const conversations = new Map<string, any[]>();
```

which stores the chat history of every conversation.

**We also have two statements:**

Getting chat history:

```ts
const chatHistory = conversations.get(conversationId) || [];
```

Saving chat history:

```ts
conversations.set(conversationId, chatHistory);
```

- Both of these perform data access.
- Therefore, they belong in the **Repository layer**, not inside `index.ts`.

## Creating a Repository

Create a new file:

```text
server/repositories/conversation.repository.ts
```

Inside it:

```ts
// conversationId -> chatHistory
const conversations = new Map<string, any[]>();
```

- This way, we are storing data **in memory**.
- This is simply **one implementation** of storing conversations.
- Tomorrow we might store data in a database.
- The rest of the application should not care.

This is called an **implementation detail**.

### Don't Export Implementation Details

Avoid this:

```ts
export const conversations = new Map<string, any[]>();
```

- Doing this exposes the internal implementation of the repository.
- Instead, expose only what other modules need.

### Public Interface

- Every module should expose only its **public interface**.
- Everything else should remain private.

For our repository, the public interface consists of two operations:

1. Get chat history
2. Save chat history

```ts
export function getChatHistory(conversationId: string) {
    return conversations.get(conversationId) || [];
}

export function setChatHistory(conversationId: string, chatHistory: any[]) {
    conversations.set(conversationId, chatHistory);
}
```

### Analogy: TV Remote

Think of a TV remote.

### Outside

- Buttons
- Power button
- Volume button
- Channel button

These form the **public interface**.

### Inside

- Circuit board
- Chips
- Wires
- Battery connections

These are **implementation details**.

- Users don't care how the remote works internally.
- They only care about the buttons.

A module should be designed the same way.

### Benefits

With this design:

- `index.ts` does not know where conversations are stored.
- Storage can change without affecting other modules.

No changes are required in the rest of the application because it only depends on `getChatHistory()` and `setChatHistory()`.

### Problem with Standalone Functions

- Although this works, there is a small design issue.
- These look like generic utility functions.
- It is not immediately obvious that they belong to the **Conversation Repository**.

### Better Approach: Export an Object

- Instead of exporting individual functions, export an object.

```ts
export const conversationRepository = {
    getChatHistory(conversationId: string) {
        return conversations.get(conversationId) || [];
    },

    setChatHistory(conversationId: string, chatHistory: any[]) {
        conversations.set(conversationId, chatHistory);
    },
};
```

## Using the Repository

Now in `index.ts`:

```ts
import { conversationRepository } from "./repositories/conversation.repository";
```

Getting chat history:

```ts
const chatHistory = conversationRepository.getChatHistory(conversationId);
```

Saving chat history:

```ts
conversationRepository.setChatHistory(conversationId, chatHistory);
```

This makes it immediately clear that:

> We are asking the **Conversation Repository** to retrieve and store conversation data.
