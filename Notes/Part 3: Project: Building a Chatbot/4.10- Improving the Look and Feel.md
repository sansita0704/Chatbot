# Improving the Chatbot UI & UX

## 1. Push the Form to the Bottom of the Screen

- Make the chatbot a vertical flex container.

Parent container:

```tsx
<div className="flex flex-col">
```

Message container:

```tsx
<div className="flex-1">
```

### Why `flex-1`?

- `flex-1` makes the message container grow and occupy all available space inside the flex container.

### Why Nothing Happened Initially?

- The chatbot component had no defined height.
- As a result `flex-1` had no extra space to grow into.

### Temporary Fix: Use Full Screen Height

- Add `h-screen` to the chatbot container.

```tsx
<div className="flex flex-col h-screen">
```

### Why `h-screen` Is Not Ideal

- Hardcoding height inside the component makes it inflexible.
- Different pages may need different chatbot sizes:

Full-page chatbot:

```text
Entire screen
```

Sidebar chatbot:

```text
Small sidebar section
```

Therefore:

> Component size should be controlled from outside the component.

### Move Height Control to Parent Component

- Remove `h-screen` from `ChatBot`.
- Apply it where `ChatBot` is used: `<App />`.

Example:

```tsx
<div className="p-4 h-screen">
    <ChatBot />
</div>
```

### New Problem After Moving Height

- Even though App had `h-screen`, the chatbot still didn't fill the available height.
- This is because HTML elements do not automatically stretch to fill their parent's height.

### Fix: Set ChatBot Height to 100%

Inside ChatBot:

```tsx
className = "h-full";
```

Result:

- Layout works again.
- Form stays at the bottom.

### Width Should Also Be Controlled Externally

- Just like height, width should not be hardcoded inside ChatBot.
- Width should be controlled by the parent.

Apply width where ChatBot is used:

```tsx
<div className="p-4 h-screen w-2xl">
    <ChatBot />
</div>
```

- Later it can easily be changed `w-4xl` or any other width.
- This makes the component reusable and flexible.

## 2. Improving Scrolling Behavior

- Currently, after several messages, the entire page becomes scrollable.
- The input area disappears from view.
- Users must scroll back down to type.

Poor user experience.

### Desired Behavior

- Only the message area should scroll.
- The input area should always remain visible.

### Add Scrolling to Message Container

Message container:

```tsx
<div className="overflow-y-auto">
```

### Meaning

If content exceeds the container height:

- Scrollbar appears automatically.
- Only this container scrolls.

Result:

- Messages scroll.
- Input remains fixed at bottom.

## 3. Fixing Auto Scrolling

Previously:

```tsx
formRef.scrollIntoView();
```

was used.

### Why It Worked Before

- The form was moving.

### Why It Doesn't Work Now

- The form is always visible.
- Scrolling to it does nothing.

### Better Approach

- Automatically scroll to the last message instead.

Previously:

```tsx
useRef<HTMLFormElement>();
```

Now:

```tsx
useRef<HTMLDivElement>();
```

- Rename `formRef` to `lastMessageRef`.

Before:

```tsx
<form ref={formRef}>
```

Remove this ref completely.

### Assign Ref to Last Message

While rendering messages:

```tsx
ref={
  index === messages.length - 1
    ? lastMessageRef
    : null
}
```

## 4. Paste Validation Bug

- After pasting text in form, the submit button remained disabled.
- Validation was not triggered correctly.

### Root Cause

- The form reset logic was incomplete.
- Earlier `reset()` was used.

### Fix

Provide explicit default values.

```tsx
reset({ prompt: "" });
```

### Why?

- React Hook Form needs the field reset to a known value.
- The available properties come from:

```tsx
type FormData = {
    prompt: string;
};
```

## 5. Auto Focus

- After refreshing the page, textarea is not focused.
- User must click inside it before typing.

### Solution

Add `autoFocus` to the textarea.

```tsx
<textarea autoFocus />
```
