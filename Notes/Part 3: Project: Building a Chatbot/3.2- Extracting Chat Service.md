# Chat Service

Controllers are responsible for:

- Receiving HTTP requests
- Validating requests

Once validation succeeds, they delegate the actual work to a **Service**.

A service should know **nothing** about:

- HTTP requests
- HTTP responses
- Express

## Identifying Business Logic

Inside our route handler we have:

```ts
const { prompt, conversationId } = req.body;
```

- This works directly with the HTTP request.
- Therefore, it belongs to the **Controller**.

At the end we have:

```ts
res.json({
    message: response.text,
});
```

- This works with the HTTP response.
- Therefore, it also belongs to the **Controller**.
- Everything between these two pieces of code is application logic.
- That logic belongs to the **Service**.

## Creating Chat Service

Create:

```text
server/services/chat.service.ts
```

```ts
import { GoogleGenAI } from "@google/genai";
import { conversationRepository } from "../repositories/conversation.repository";

// Implementation detail
const client = new GoogleGenAI({
    apiKey: process.env.GEMINI_API_KEY,
});

export const chatService = {
    async sendMessage(prompt: string, conversationId: string) {
        const chatHistory =
            conversationRepository.getChatHistory(conversationId);

        chatHistory.push({
            role: "user",
            parts: [{ text: prompt }],
        });

        const response = await client.models.generateContent({
            model: "gemini-2.5-flash-lite",
            contents: chatHistory,
            config: {
                temperature: 0.2,
                maxOutputTokens: 100,
            },
        });

        chatHistory.push({
            role: "model",
            parts: [{ text: response.text }],
        });

        conversationRepository.setChatHistory(conversationId, chatHistory);

        return response;
    },
};
```

### Why Move `client` Here?

Previously we had:

```ts
const client = new GoogleGenAI(...)
```

inside `index.ts`.

- Now it is inside `chat.service.ts`.
- This is because, the AI model being used is an **implementation detail**.

Only the service should know:

- Which LLM is used
- How prompts are sent
- How responses are received

Other modules should remain unaware.

### Benefit

- Suppose today we use "Gemini".
- Later we switch to some other AI.
- Only **`chat.service.ts`** needs to change.

No other module is affected.

### Public Interface

Other modules simply use:

```ts
chatService.sendMessage(...)
```

- They don't know what happens internally.
- This is **abstraction**.

## Problem: Leaky Abstraction

- In `index.ts` we use `response.text`.
- The property `text` is specific to **Gemini**.
- Another LLM might use some other property.
- This means Gemini-specific details are leaking outside the service.

### What is a Leaky Abstraction?

- The service is supposed to hide Gemini.
- However, it still exposes Gemini's response object.
- Therefore, the abstraction is incomplete.

This is called a **Leaky Abstraction**.

> A leaky abstraction is an abstraction that exposes some of its internal implementation details to its consumers.

### Solution

- Create a response type that is independent of any LLM in the service.

```ts
type ChatResponse = {
    id: string | undefined;
    message: string | undefined;
};
```

- This type represents a response from an LLM.

### Updated Service

```ts
export const chatService = {
    async sendMessage(
        prompt: string,
        conversationId: string,
    ): Promise<ChatResponse> {
        const chatHistory =
            conversationRepository.getChatHistory(conversationId);

        chatHistory.push({
            role: "user",
            parts: [{ text: prompt }],
        });

        const response = await client.models.generateContent({
            model: "gemini-2.5-flash-lite",
            contents: chatHistory,
            config: {
                temperature: 0.2,
                maxOutputTokens: 100,
            },
        });

        chatHistory.push({
            role: "model",
            parts: [{ text: response.text }],
        });

        conversationRepository.setChatHistory(conversationId, chatHistory);

        return {
            id: response.responseId,
            message: response.text,
        };
    },
};
```

- Now the service returns a generic response instead of a Gemini-specific one.
- If we change LLMs later, only this file needs modification.

So, the chat service module encapsulates all the details of working with the LLM and exposes a simple interface through its `sendMessage()` method.
