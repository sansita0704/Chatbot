# Recap

## Breaking the Chatbot into Modules

Instead of putting all code into one large component, the chatbot was divided into smaller modules.

Each module has a single responsibility.

| Module          | Responsibility                       |
| --------------- | ------------------------------------ |
| ChatBot         | Manages the conversation             |
| ChatInput       | Handles user input and form behavior |
| ChatMessages    | Displays messages                    |
| TypingIndicator | Shows loading/typing state           |

### Keeping Internal Logic Private

Each module:

- Hides its internal implementation details.
- Exposes only a clean public interface.

This makes components easier to use and easier to change later.

### Abstraction

In the chatbot:

- Components expose only what other components need.
- Internal details remain hidden.
- Other parts of the app do not depend on those details.

This concept is called abstraction.

### Why This Matters

Because changes become safer.

For example:

- Change how ChatInput works internally.
- Change how messages are rendered.
- Optimize performance.
- Switch implementation details.

As long as the public interface stays the same, the rest of the application keeps working.

### Minimizing the Impact of Changes

- The goal is to minimize the impact of changes.
- Instead of one change breaking many files, changes remain localized to a small module.

### Why Large Teams Need This

- These design decisions allow teams to build large, complex systems without everything falling apart whenever something changes.
