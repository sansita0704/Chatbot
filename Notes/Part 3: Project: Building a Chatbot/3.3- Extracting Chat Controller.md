# Chat Controller

The controller acts as the entry point to our application.

Its responsibilities are:

- Receive HTTP requests
- Validate request data
- Call one or more services
- Return HTTP responses

## Creating Chat Controller

Create:

```text
server/controllers/chat.controller.ts
```

```ts
import type { Request, Response } from "express";
import { chatService } from "../services/chat.service";
import z from "zod";

const chatSchema = z.object({
    prompt: z
        .string()
        .trim()
        .min(1, "Prompt is required.")
        .max(1000, "Prompt is too long (max 1000 characters)"),

    conversationId: z.string().uuid(),
});

export const chatController = {
    async sendMessage(req: Request, res: Response) {
        const parseResult = chatSchema.safeParse(req.body);

        if (!parseResult.success) {
            res.status(400).json(parseResult.error.format());
            return;
        }

        try {
            const { prompt, conversationId } = req.body;

            const response = await chatService.sendMessage(
                prompt,
                conversationId,
            );

            res.json({
                message: response.message,
            });
        } catch (error) {
            res.status(500).json({
                error: "Failed to generate a response.",
            });
        }
    },
};
```

### Implementation Detail

- Inside this controller `chatSchema` is an implementation detail.

The rest of the application should not know:

- We use Zod
- How validation works

It only needs the controller's public interface.

### Public Interface

```ts
chatController.sendMessage();
```

- That is the only thing other modules (its consumers) should use.
- If we replace Zod with another validation library later, only this controller changes.

### Updating `index.ts`

```ts
app.post("/api/chat", chatController.sendMessage);
```

- `index.ts` no longer contains business logic.
- It simply registers routes.
