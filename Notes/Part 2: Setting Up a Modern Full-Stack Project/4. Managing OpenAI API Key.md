# Environment Variables

- The correct way to manage API keys is by using **Environment Variables**.
- An **environment variable** is a variable stored at the operating system level that can be accessed by applications during runtime.

## Creating an Environment Variable

To create an environment variable in the terminal:

```bash
export <VARIABLE_NAME>=<VALUE>
```

### Example

```bash
export GEMINI_API_KEY=1234
```

### Accessing Environment Variables

- Environment variables can be accessed through `process.env`.

Example:

```ts
app.get("/", (req: Request, res: Response) => {
    res.send(process.env.GEMINI_API_KEY);
});
```

Run the server:

```bash
bun run dev
```

If the environment variable exists, visiting the endpoint will display:

```text
1234
```

### Problem with This Approach

- Although this works, it is not practical.
- Every time we start the application, we would need to manually define all environment variables again.

- This becomes tedious, especially as the number of variables grows.

## Better Approach: Using dotenv

- To simplify environment variable management, we use the **dotenv** package.

- `dotenv` automatically loads environment variables from a `.env` file into `process.env`.

### Install dotenv

Inside the server directory:

```bash
bun add dotenv
```

- Create a file named `.env` inside the server directory.
- Add your variables:

```env
GEMINI_API_KEY=abcd
```

### Load Environment Variables

At the very top of your application:

```ts
import dotenv from "dotenv";
dotenv.config();
```

#### Why at the Top?

```ts
dotenv.config();
```

must execute before any code tries to access environment variables.

When this function runs:

1. The `.env` file is read.
2. All variables inside it are loaded into `process.env`.
3. The rest of the application can access them normally.

### Why Doesn't the Value Change?

- Environment variables already present in the terminal session take precedence over values from `.env`.
- The value defined using `export` overrides the value inside `.env`.

### Fix

Remove the existing environment variable:

```bash
unset GEMINI_API_KEY
```

- Then restart the server.
- Now the value from `.env` will be used.

## Problem

- If we push our project to GitHub, other developers can see the code but have no way of knowing which environment variables are required.

### Solution: `.env.example`

Create a file named:

```text
.env.example
```

and list all required variables without their actual values.

Example:

```env
GEMINI_API_KEY=
```

### Important Note

If you modify the `.env` file while the server is running:

```env
GEMINI_API_KEY=new-value
```

the changes will **not** be picked up automatically.

- Even if Bun is running in watch mode, it does not automatically reload environment variables.
- Therefore, after modifying `.env`, you must manually restart the server.
