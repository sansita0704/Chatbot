# Husky

- We can manually run `bun run format` before committing.
- However, people often forget.
- To automate this process, we use **Husky**.

## What is Husky?

- **Husky** helps automate our Git workflow.
- It allows us to run certain commands, such as formatting our code or running tests, before committing code.

## Install Husky

Documentation:

```text
https://typicode.github.io/husky/get-started.html
```

Install in the root directory:

```bash
bun add --dev husky
```

Initialize Husky:

```bash
bunx husky init
```

This command:

1. Creates a `.husky` directory.
2. Creates a default `pre-commit` script.

## Running Prettier Before Every Commit

A simple solution would be to add:

```bash
bun run format
```

inside the pre-commit hook.

- However, this introduces two problems.

### Problem 1

- It formats the entire codebase.
- As the project grows, this becomes slower.

### Problem 2

- It may modify files unrelated to our current feature.

Example:

We changed only:

```text
App.tsx
Button.tsx
```

but Prettier reformats dozens of unrelated files.

- These files then appear in the commit even though we never touched them.

## Better Solution: lint-staged

- Instead of formatting the entire project, we should only format files that are currently staged for commit.
- For this, we use **lint-staged**.
- lint-staged runs commands only on files that have been staged.

Documentation:

```text
https://www.npmjs.com/package/lint-staged
```

### Install lint-staged

```bash
bun add -d lint-staged
```

### Update the Pre-Commit

Inside `.husky/pre-commit`:

```bash
bunx lint-staged
```

Now every commit will trigger lint-staged.

### Configure lint-staged

Create `.lintstagedrc` file:

```json
{
    "*.{js,jsx,ts,tsx}": "prettier --write"
}
```

### Understanding the Configuration

#### Key: file pattern

```json
"*.{js,jsx,ts,tsx}"
```

Matches any staged file with one of these extensions:

- `.js`
- `.jsx`
- `.ts`
- `.tsx`

#### Value: command

```json
"prettier --write"
```

Command to execute on matching staged files.

### Why Not Use This?

```bash
prettier --write .
```

Because `.` means:

> Start formatting the entire project.

- But lint-staged automatically passes only the staged file names to the command.
- Therefore `prettier --write is sufficient.

## Complete Workflow

1. Modify files
2. Stage them
3. Commit
4. Husky automatically runs:

```bash
bunx lint-staged
```

5. lint-staged checks `.lintstagedrc`:

```json
{
    "*.{js,jsx,ts,tsx}": "prettier --write"
}
```

6. Prettier formats only the staged files.
