# Controlling the Output Format

- When building real-world applications with LLMs, it's not enough to control **what** the model says, we also need to control **how** it says it.

Depending on the application, we may want the response as:

- Plain text
- A bullet list
- Markdown
- A structured JSON object

## 1. Default Output

- By default, if we ask the model a question, it usually responds with a **paragraph of plain text**.
- For many use cases, this is perfectly acceptable because it is natural and readable.

However, we can still control various aspects of plain text responses.

### Controlling Response Length

Specifying the desired response length is important for:

- Better user experience
- Reducing API costs

### Why?

If the model is asked to summarize a long piece of text without any length constraint, it may generate a lengthy response that:

- Uses more output tokens
- May exceed the maximum output token limit

### Better Prompt Examples

```text
Summarize this in two short sentences.
```

or

```text
Summarize this in under 100 words.
```

### Ensure Complete Responses

When limiting the response length, it's also good practice to instruct the model to finish naturally.

### Example

```text
Make sure the summary is complete and doesn't end mid-sentence.
```

## 2. Markdown

- For more polished responses, ask the model to return the output in **Markdown**.

Markdown allows formatting such as:

- Bold text
- Italics
- Bullet points
- Other basic formatting

### Example

```text
Summarize this review in two bullet points using Markdown.
Highlight important details in bold.
```

## 3. Comma-Separated List

For tasks like extracting keywords or tags, a comma-separated list is often useful.

### Example

```text
Extract three keywords that describe this article.
Return them as a comma-separated list in lowercase.
```

The more specific the formatting instructions, the better the output.

## 4. JSON

When an application needs structured data, request the output as **JSON**.

### Example

```text
From the paragraph below, extract all product names and their prices.

Return a valid JSON array of objects.
Each object should include:

- name (string)
- price (number without currency symbols)
```

This produces structured data that can be easily processed in code.

### Avoid Extra Explanations

- Language models sometimes include explanations along with the requested JSON.
- In applications, this extra text is often undesirable.
- To prevent this, explicitly instruct the model to return **only JSON**.

### Example

```text
Only return valid JSON. No explanation or extra text.
```

This increases the likelihood of receiving clean JSON that can be parsed directly by the application.
